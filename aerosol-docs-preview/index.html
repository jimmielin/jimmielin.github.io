CAM Abstract Aerosol Interface

Abstract Aerosol Interface (AAI) — Developer Documentation

The Abstract Aerosol Interface provides a polymorphic Fortran interface that decouples physical processes (radiation, microphysics, ice nucleation, heterogeneous freezing, wet/dry deposition) from the details of any particular aerosol representation.

Three abstract base classes define the contract:

Base class Purpose Module
aerosol_properties Time-invariant configuration: bins, species, naming, activation fractions aerosol_properties_mod
aerosol_state Time-varying fields: mixing ratios, number concentrations, derived quantities aerosol_state_mod
aerosol_optics Optical property calculation (SW extinction/absorption/albedo/asymmetry, LW absorption) aerosol_optics_mod

Concrete implementations

Each aerosol model extends the three base classes:

Aerosol model Properties State Optics dispatch
Modal (MAM) modal_aerosol_properties modal_aerosol_state via aerosol_optics_core
Bulk bulk_aerosol_properties bulk_aerosol_state via aerosol_optics_core
CARMA carma_aerosol_properties carma_aerosol_state via aerosol_optics_core

Seven optics strategies implement aerosol_optics, selected per-bin by the opticstype string from each species' physical-property file:

  • hygro_aerosol_optics — hygroscopic growth (standard MAM modes)
  • hygroscopic_aerosol_optics — hygroscopic (legacy variant)
  • hygrocoreshell_aerosol_optics — core-shell Mie (black carbon)
  • hygrowghtpct_aerosol_optics — weight-percent lookup (sulfate)
  • refractive_aerosol_optics — volume-mixing refractive index
  • insoluble_aerosol_optics — no water uptake (dust, some primary)
  • volcrad_aerosol_optics — volcanic aerosol (prescribed)

Infrastructure modules

  • aerosol_optics_core — factory + per-bin SW/LW optics driver
  • aerosol_instances_mod — lifecycle manager for properties/state objects
  • radiative_aerosol — façade providing query routines (rad_aer_get_info, rad_aer_get_props, etc.)
  • radiative_aerosol_definitions — shared constants, types, and namelist parsing

Where is the AAI called from?

The abstract interface is consumed by physics parameterizations throughout CAM. Key callers include:

  • Radiation (aer_rad_props_sw, aer_rad_props_lw in src/physics/cam/) — uses aerosol_optics for SW/LW optical properties
  • Microphysics (nucleate_ice, hetfrz) — uses aerosol_state for ice nucleation and heterogeneous freezing
  • Activation (ndrop, ndrop_bam) — uses aerosol_properties for CCN activation
  • Wet deposition (aero_wetdep_cam) — uses aerosol_state for scavenging
  • Dry deposition (aer_drydep_mod) — uses aerosol_properties for settling
  • Convective processing (aero_convproc) — uses aerosol_state for in-cloud processing
  • Chemistry (mo_setsox) — uses aerosol_state for aqueous SO₂ oxidation

If I change something, what is affected?

  • Changing an abstract type's interface (adding/modifying a deferred procedure) requires updating all concrete implementations (modal, bulk, CARMA).
  • Changing a concrete implementation affects only the physics of that aerosol model but is exercised by all callers that use the abstract interface.
  • Changing optics affects radiation only — the optics types are constructed per-bin inside aerosol_optics_core and consumed by the radiation code.
  • Changing radiative_aerosol or radiative_aerosol_definitions affects initialization and namelist parsing — these are called early in the model setup and feed all downstream objects.
